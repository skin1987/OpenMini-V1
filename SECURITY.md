## 🔒 安全政策声明

OpenMini 团队高度重视项目和用户的安全。我们承诺：

- ✅ **主动安全**: 在开发生命周期的每个阶段都考虑安全性
- ✅ **快速响应**: 对报告的安全漏洞在 48 小时内响应
- ✅ **透明公开**: 及时发布安全公告和修复方案
- ✅ **持续改进**: 定期进行安全审计和依赖更新
- ✅ **负责任披露**: 尊重并保护安全研究者的隐私

### 我们的安全承诺

1. **定期安全扫描**: 通过 CI/CD 管道每日自动执行依赖漏洞扫描和 Secret 检测
2. **及时修复**: 对于 CRITICAL 和 HIGH 级别漏洞，在确认后 72 小时内发布补丁
3. **安全公告**: 通过 GitHub Security Advisories 发布安全更新通知
4. **最小权限原则**: 系统组件遵循最小权限设计

## 🦀 支持版本

OpenMini 支持以下 Rust 版本：

### 当前支持

| Rust 版本 | 状态 | 支持截止日期 | 说明 |
|-----------|------|-------------|------|
| **Stable (最新)** | ✅ 完全支持 | 持续支持 | 推荐用于生产环境 |
| **Beta** | ⚠️ 实验性支持 | 下一个 Stable 发布后 | 用于提前测试新特性 |

### 版本策略

- **MSRV (Minimum Supported Rust Version)**: 1.75.0
- 我们保证在 MSRV 及以上版本上能够正常编译和运行
- 每次升级 MSRV 将在 CHANGELOG 中说明原因
- 建议始终使用最新的 Stable 版本以获得最佳安全性和性能

### 不支持的版本

- ❌ Nightly 版本（除非明确标注）
- ❌ 低于 MSRV 的任何版本
- ❌ 已 EOL (End of Life) 的 Rust 版本

## 🐛 漏洞报告流程

我们发现并鼓励负责任的安全研究。如果您发现了安全漏洞，请按照以下流程报告：

### 报告方式（按优先级排序）

#### 方式 1: 私密报告（推荐）

**邮箱地址:** `security@openmini.ai`

**邮件主题格式:**
```
[Security Report] <漏洞简要描述> - <影响范围>
```

**邮件内容应包含:**
1. 漏洞描述和复现步骤
2. 影响评估（CVSS 评分如可能）
3. 受影响的版本范围
4. 建议的修复方案（可选）
5. 您的联系信息（用于后续沟通）

#### 方式 2: GitHub Security Advisory（推荐）

通过 GitHub 的私有安全报告功能提交：

1. 访问仓库页面
2. 点击 **Security > Security advisories**
3. 选择 **Draft a new security advisory**
4. 填写详细信息

**优势:**
- 完全私密，只有仓库维护者可见
- 与 GitHub Issues 和 PR 无缝集成
- 支持 CVE 分配

#### 方式 3: GitHub Issue（仅限低风险）

对于非紧急的低风险安全问题，可以使用 Issue 模板：

- 标题前缀: `[Security]`
- 选择标签: `security`
- **注意:** 请勿在 Issue 中暴露完整的漏洞利用代码

### 报告处理时间线

| 时间节点 | 行动 |
|---------|------|
| **24 小时内** | 确认收到报告，分配处理人员 |
| **48 小时内** | 初步评估和分类 |
| **72 小时内** | 提供初步修复计划或请求更多信息 |
| **7 天内** | 完成 CRITICAL/HIGH 级别漏洞的修复 |
| **30 天内** | 公开安全公告（协调披露） |

### 协调披露政策

我们遵循 **90天协调披露** 原则：

1. **第 0 天**: 收到漏洞报告
2. **第 0-7 天**: 确认漏洞，开发修复方案
3. **第 7-90 天**: 用户有足够时间更新（除非漏洞已被主动利用）
4. **第 90 天**: 公开披露（即使修复尚未完成）

**例外情况:**
- 如果漏洞已被广泛利用，将立即公开并发布紧急修复
- 如果研究者希望更早披露，我们可以协商调整时间线

### 奖励与致谢

- 所有有效漏洞报告将在安全公告中致谢（除非您要求匿名）
- 对于重大漏洞发现者，我们将提供：
  - OpenMini 贡献者证书
  - 项目文档中的特别致谢
  - 优先参与未来安全审计的机会

## ⏰ 安全更新时间线

### 补丁发布周期

| 严重程度 | 响应时间 | 修复时间 | 发布方式 |
|---------|---------|---------|---------|
| **CRITICAL** (关键) | 24 小时 | 72 小时 | 热修复 + 安全公告 |
| **HIGH** (高) | 48 小时 | 7 天 | 小版本更新 |
| **MEDIUM** (中) | 1 周 | 14 天 | 小版本更新 |
| **LOW** (低) | 2 周 | 30 天 | 大版本更新 |

### 维护窗口

- **LTS (长期支持) 版本**: 安全修复至少支持 12 个月
- **常规版本**: 安全修复支持至下一个主版本发布后 6 个月
- **EOL (终止支持)**: 不再提供安全更新

### 通知渠道

当发布安全更新时，我们将通过以下渠道通知：

1. **GitHub Releases**: 附带安全更新说明
2. **GitHub Security Advisories**: 详细的技术细节
3. **CHANGELOG.md**: 版本更新记录
4. **邮件列表** (如有): 订阅用户通知

## ⚠️ 已知限制

### 当前版本 (v1.1.0) 的安全注意事项

#### 1. 依赖项风险

OpenMini 使用了多个第三方 crate，其中部分可能存在已知或未知的安全风险：

**高风险依赖:**
- `candle-core`, `candle-transformers`: AI 框架核心库，需关注上游安全更新
- `tokio`: 异步运行时，历史上曾出现安全修复

**缓解措施:**
- ✅ 每日自动执行 `cargo audit` 扫描
- ✅ 锁定依赖版本 (`Cargo.lock`)
- ✅ 定期审查依赖更新

#### 2. 网络服务安全

- **gRPC 服务**: 默认监听端口可能暴露在公网
- **HTTP API**: 需要适当的认证和速率限制
- **管理面板**: 需要 HTTPS 和强认证机制

**建议:**
- 生产环境部署时使用反向代理 (Nginx/Traefik)
- 启用 TLS 加密通信
- 配置防火墙规则限制访问

#### 3. 数据存储

- **内存数据库**: 默认使用内存存储，数据不持久化
- **PostgreSQL 集成**: 需要正确配置连接参数和权限

**建议:**
- 不要在生产环境使用默认凭据
- 定期备份数据库
- 启用数据库审计日志

#### 4. GPU/Metal 计算

- CUDA 和 Metal 内核执行可能存在侧信道攻击风险
- 第三方模型文件 (GGUF) 可能包含恶意代码

**建议:**
- 仅从可信来源获取模型文件
- 在沙箱环境中运行不受信任的模型
- 监控 GPU 资源使用情况

#### 5. 日志和监控

- 日志可能包含敏感信息（错误堆栈、配置详情）
- Prometheus 指标端点需要访问控制

**建议:**
- 配置日志脱敏规则
- 限制指标端点的访问权限
- 实施集中式日志管理

### 功能限制

- ❌ 目前不支持内置的加密存储
- ❌ 不提供内置的身份认证服务（依赖外部实现）
- ❌ 模型加载缺少完整性校验（SHA256）
- ❌ 不支持审计日志的防篡改机制

### 未来改进计划

- [ ] 集成 HashiCorp Vault 进行密钥管理
- [ ] 实现模型文件的签名验证
- [ ] 添加 RBAC (基于角色的访问控制)
- [ ] 支持 FIPS 140-2 合规模式
- [ ] 实现零信任网络架构

## 🛡️ 安全最佳实践

### 部署清单

在部署 OpenMini 到生产环境前，请确保：

#### 基础设施层

- [ ] 使用最新的操作系统和安全补丁
- [ ] 配置防火墙，只开放必要端口
- [ ] 启用 SELinux/AppArmor (Linux)
- [ ] 使用专用服务账户运行（不要 root）
- [ ] 配置资源限制 (cgroups/namespaces)

#### 应用层

- [ ] 修改所有默认密码和密钥
- [ ] 启用 HTTPS/TLS 加密
- [ ] 配置 CORS 策略
- [ ] 设置速率限制和请求大小限制
- [ ] 启用请求日志和审计跟踪

#### 数据层

- [ ] 使用强密码策略
- [ ] 配置数据库备份策略
- [ ] 加密敏感配置字段
- [ ] 实施数据保留和删除策略

#### 监控层

- [ ] 配置异常行为告警
- [ ] 监控登录失败尝试
- [ ] 跟踪 API 响应时间和错误率
- [ ] 设置磁盘空间和内存使用告警

### 代码安全规范

开发者应遵循：

1. **输入验证**: 所有外部输入必须验证和消毒
2. **错误处理**: 不要在错误消息中泄露内部信息
3. **依赖管理**: 定期更新依赖，审查新版本的变更
4. **Secret 管理**: 永远不要将密钥提交到代码库
5. **日志安全**: 避免在日志中记录敏感数据
